# DMS_RB Database Schema

This document describes the SQLite schema used by the active `DMS_RB` application.

Database file:
- `DMS_RB/data/dms.db`

Schema source:
- `DMS_RB/server/db/schema.sql`

## Overview

The database is split into four areas:

1. Document data
2. User and notification data
3. Admin configuration data
4. Workflow approver rules

The main record is `dms_header`. Most other document tables hang off `document_id`.

## Core Relationships

- `dms_header.document_id` -> parent document record
- `dms_object_links.document_id` -> `dms_header.document_id`
- `dms_attachments.document_id` -> `dms_header.document_id`
- `dms_workflow_steps.document_id` -> `dms_header.document_id`
- `dms_change_history.document_id` -> `dms_header.document_id`

Foreign keys on those child tables are defined with `ON DELETE CASCADE`.

## Tables

### `dms_header`

Primary document header and flattened search/filter fields.

Important columns:
- `document_id` - unique business ID for the document
- `rig`
- `doc_type`
- `doc_group`
- `status`
- `originator`
- `originator_username`
- `created_date`
- `last_modified`
- `version`
- `doc_date`, `manu_name`, `manu_serial`, `alert_num`, `cert_auth`, `cert_num`, `add_desc`, `doc_loc`
- `search_text`
- `ocr_text`

Indexes:
- `idx_header_status`
- `idx_header_originator`
- `idx_header_rig`
- `idx_header_doc_group`

### `dms_object_links`

Stores linked work order / equipment style references for a document.

Important columns:
- `document_id`
- `work_order`
- `work_order_description`
- `owning_department_id`
- `em_asset_number`
- `equipment_text`
- `equipment_description`
- `rig`
- `parent`

Index:
- `idx_links_doc`

### `dms_attachments`

Stores attachment metadata for uploaded files.

Important columns:
- `document_id`
- `filename`
- `original_name`
- `file_path`
- `file_size`
- `mime_type`
- `uploaded_by`
- `uploaded_date`

Index:
- `idx_attach_doc`

### `dms_workflow_steps`

Stores workflow state per document step.

Important columns:
- `document_id`
- `step_number`
- `step_name`
- `step_key`
- `status`
- `assigned_approvers` - JSON array
- `actioned_by`
- `actioned_by_name`
- `action_date`
- `rejection_reason`

Index:
- `idx_workflow_doc`

### `dms_change_history`

Immutable history / audit trail for document changes.

Important columns:
- `document_id`
- `version`
- `changed_by`
- `changed_by_name`
- `change_date`
- `change_type`
- `change_summary`
- `previous_data` - JSON snapshot

Index:
- `idx_history_doc`

### `dms_notifications`

User notifications generated by workflow and document actions.

Important columns:
- `id` - primary key
- `recipient_username`
- `type`
- `message`
- `document_id`
- `link`
- `read`
- `created_at`

Index:
- `idx_notif_user`

### `users`

Application users and roles.

Important columns:
- `username` - unique login key
- `display_name`
- `email`
- `department`
- `role`
- `active`
- `created_date`
- `last_login`

### `config_doc_types`

Admin-managed document type master data.

Important columns:
- `code` - unique
- `description`
- `active`

### `config_doc_groups`

Admin-managed document group master data.

Important columns:
- `doc_type`
- `code` - unique
- `description`
- `workflow_required`
- `active`

### `config_field_visibility`

Visibility rules per document group and field.

Important columns:
- `doc_group`
- `field_name`
- `visibility`

Constraint:
- unique on `doc_group, field_name`

### `config_attachment_naming`

Attachment naming field order per document group.

Important columns:
- `doc_group`
- `field_order`
- `field_name`

Constraint:
- unique on `doc_group, field_order`

### `config_work_centers`

Department / work center reference data.

Important columns:
- `department_id` - unique
- `department_name`
- `description`
- `active`

### `config_email`

SMTP and email behavior settings.

Important columns:
- `setting_key` - unique
- `setting_value`

### `workflow_approvers_discipline`

Approver rules for discipline and E&M approvals by department.

Important columns:
- `department_id`
- `approval_type` - typically `msv` or `em`
- `approver_username`
- `active`

Constraint:
- unique on `department_id, approval_type, approver_username`

### `workflow_approvers_maintenance`

Approver rules for maintenance-based approval routing.

Important columns:
- `maintenance_strategy`
- `maintenance_days`
- `approval_type`
- `approver_username`
- `active`

## Notes

- Most admin tables use soft deletes via the `active` column.
- Workflow step assignees are stored as JSON in `dms_workflow_steps.assigned_approvers`.
- Discipline approver rules support multiple approvers by storing one row per approver for the same department and approval type.
- Maintenance approver rules also support multiple approvers by storing one row per approver for the same strategy/days/type combination.
